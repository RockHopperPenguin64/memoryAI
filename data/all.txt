費やした時間は2時間。
原因は根本的な仕組みの方にありました。
最初はプレイヤーの位置にtargetPositionというインスタンス化された時の位置を保存する役割のあるVector3型の変数があるのですが、それが原因だったので、offsetという変数を作り、プレイヤーとの距離を計算してそこからプレイヤーのGlobalPositionを足して位置関係を保つという仕組みに変えました。
仕組み自体は限りなく正解に近いのですが、欠点があります。
そう、プレイヤーが中心とすると0,0,0にインスタンス化された後すぐに行ってしまうのです。
理由は簡単です。
PowerStarがインスタンス化された後、プレイヤーの方で1フレーム後に場所を移動させるのですが、その移動させる前のフレームで場所とプレイヤーの関係を記録していたのがネックでした。
なのでプレイヤーが持っているtargetPositionというスターをどこに配置させるかを司る変数、人でいう脳ミソまたは心臓的な役割の変数ですね。
それをstar.csで参照させてその変数とプレイヤーの位置で関係を維持するようにしたら安定するようになりました。
それで終わりではないです、試しにビルドしてやっていたら遠くなるほどスターが画面から消えていくと言う現象が起こりました。
それの原因を究明すべく、私お手製のパラメーターを変えるだけで位置を変えられる、しかもビルド中も可能なカメラの方を使い原因を究明しました。
まず起こっていた現象はプレイヤーとリス地（0,0,0）との距離だけプレイヤーから離れたところにStarが生成されていました。
で、そのようになっている原因はプレイヤーがわのtargetPositionには最終的にplayerPositionというプレイヤーの位置を足していたため遠くなったというのが原因でした。
そのコードを消したら直りました。
でその後、プレイヤーの向いている方向に飛ぶようにしたり、待機中は回転するようにしたり、生成される場所の微調整をしたり、飛んでいる時はチャージしている状態を検知して発車後に止まらないようにしたりと微調整をし、チャージ攻撃は90％くらい完成しました。

原因は根本的な仕組みの方にありました。
最初はプレイヤーの位置にtargetPositionというインスタンス化された時の位置を保存する役割のあるVector3型の変数があるのですが、それが原因だったので、offsetという変数を作り、プレイヤーとの距離を計算してそこからプレイヤーのGlobalPositionを足して位置関係を保つという仕組みに変えました。
仕組み自体は限りなく正解に近いのですが、欠点があります。
そう、プレイヤーが中心とすると0,0,0にインスタンス化された後すぐに行ってしまうのです。
理由は簡単です。
PowerStarがインスタンス化された後、プレイヤーの方で1フレーム後に場所を移動させるのですが、その移動させる前のフレームで場所とプレイヤーの関係を記録していたのがネックでした。
なのでプレイヤーが持っているtargetPositionというスターをどこに配置させるかを司る変数、人でいう脳ミソまたは心臓的な役割の変数ですね。
それをstar.csで参照させてその変数とプレイヤーの位置で関係を維持するようにしたら安定するようになりました。
それで終わりではないです、試しにビルドしてやっていたら遠くなるほどスターが画面から消えていくと言う現象が起こりました。
それの原因を究明すべく、私お手製のパラメーターを変えるだけで位置を変えられる、しかもビルド中も可能なカメラの方を使い原因を究明しました。

まずはCharacterBody3Dを追加していこう！

CharacterBody3Dを追加している様子 Godot使うなら絶対に知っておけ！ - CharacterBody3Dってなんだ！
CharacterBody3Dはプレイヤーキャラクターや敵キャラクターのような能動的に動く存在を制御するために設計されたNodeです。
特に「重力やジャンプ、移動」といったゲームでよくキャラクターが使う動きを簡潔に記述できるように作られているのが特徴です。

CharacterBody3Dは、物理ベースの移動を行うための"Velocity"プロパティと、衝突を考慮しながら移動を処理する"MoveAndSlide()"メソッドを標準で備えています。
これらをうまく活かすことで、重力や床との滑りなどの基本的なキャラクターの挙動が自然に再現できます。


つまり、CharacterBody3DやRegidBody3D以外のテキトーなNodeを使って動きをプログラミングするよりもCharacterBody3Dを使う方が楽に移動、ジャンプを作ることができる且挙動が自然になるのです。
まあ、最初はプレイヤー、敵を作る時に使うのはCharacterBody3Dくらいのノリで覚えておいてください。


collisionShape3Dを追加している様子
次にCollisionShape3Dを追加します。


これを知らなきゃ話にならない。
CollisionShape3Dって何？
CollisionShape3Dは、物理演算を行うための形状情報を与えるためのNodeです。
つまり、Godotにおける「当たり判定の形＝Shape」を定義するNodeです。
単体では機能せず、CharacterBody3DやRigidBody3D、Area3Dなどの物理Nodeの子として使われます。
親Nodeに"どの範囲に当たり判定を持たせるかを教える"役目を果たしています。


これがないと例えば重力を作ったとしても床をすり抜けて落ちていっちゃうとかそういうふうになります。
CharacterBody3DやRigidBody3D、Area3Dとセットで追加するというふうに思っていればOKです。


meshInstance3Dを追加している様子
最後にMeshInstance3Dを追加していきます。


これをなくしてゲームは成り立たない。
MeshInstance3Dとは何者？
MeshInstance3Dは、Godotで3Dモデルをシーン上に表示するために使うNodeです。

C#でやる場合は特に。

例えば、numberという変数に8.2を代入する場合、public float number = 8.2f;というふうに描きます。

Godotって何？最近注目の超軽量ゲーム開発エンジン！
Godot（ゴドー　私は最初ゴードットと思っていた）は無料で使えるオープンソースのゲームエンジンです。

if (IsOnFloor() && Input.IsActionJustPressed("Jump"))
        {
            direction.Y = JumpPower;

OK、なんとなくつかめました。
まずはジャンプ回数を制限するために専用の変数を作っていきます。



private int _jumpLimit = 2;
今回作るのは二段ジャンプなのでジャンプ回数の制限は2回です。


_jumpLimitの"_"ってなんだ！h3>
今作った変数の名前に "_（アンダーバー）"をつけているのには歴とした理由があるので、解説しようと思います。

C#やGodot C#では、**クラスの中で使われる「private（外からアクセスできない）変数」には"_"をつけるスタイルがよく使われます。
これは「この変数は内部用ですよ」という意味です。

よく勘違いされるのですが、_PhysicsProcessや_Readyなどの関数にも"_"がついてますが、定数を意味する"_"とは若干違い、「Godotが内部的に呼び出す特別な関数」という意味です。


ジャンプ回数を制限する変数が書けたので、条件分岐の方を変えていきます。



if ((IsOnFloor() || _jumpLimit > 0) && Input.IsActionJustPressed("Jump"))
  
論理演算子　"||"
タイトルの方で意味は書いてしまいましたが、" || "は『または』という意味の論理演算子です。
高校数学でよく嫌われているやつですね。
この『または』は少なくともどちらか一方がTrueでTrueと返します。
高校数学だと、『少なくともこのどちらか一方で』っていうのが面倒くさいんですよね。
だから対偶を考えるんです。
数学の話になりますが、命題を全て否定したのが対偶です。
例えば、「もし n² が偶数なら、n も偶数である」っていうやつの対偶は全てを否定した 「が奇数ならば、n² も奇数である」というふうですね。
全てをひっくり返したものが対偶です。

脱線してしまったので話を戻します。
" || "は『または』という意味です。
これさえ覚えておけばいいです。


これを元に私の書いた条件式を読み解くとこのようになります。

if （床の上　または　_jumpLimitが0より大きい）且つJumpが押されたら
というふうになります。


では処理の方を書いていきましょう。



if ((IsOnFloor() || _jumpLimit > 0) && Input.IsActionJustPressed("Jump"))
　　{
            direction.Y = JumpPower;
            _jumpLimit -= 1;
        }
  
これだけで二段ジャンプできるようになります。
本当ですよ！実際にやってみせましょうか？

これらの問題を解決するには、日本国内でできるかつCO2の排出量を極限まで減らしつつ、現在の電気需要、年間1兆kWに達する必要があります。
そこで私（たち）の班は主力の発電所とその発電所で足りない分を賄う即戦力の発電所を使って新たな発電所の構成を作ることで、CO2排出量を極限まで減らせると考えました。
ということで、まずは主力となる発電所の方からそれぞれ、詳細なステータスをみていきましょう。

主力として有力なのは、
（Enter）バイオマス発電です。
バイオマス発電は植物、農業廃棄物、動物の排泄物、食品廃棄物などの有機物を燃焼・分解することで電気を得る、環境への負荷が少ない発電方法です。
有機物や、熱分解することで発生する可燃性ガス、微生物で分解して発生するメタンガスを燃やすことでタービンを回し、発電します。
発電に使用するバイオマス燃料は成長過程でco2を吸収するため、燃焼時の排出量と相殺され、結果的には環境負荷を低くなります。

（Enter）しかし、この発電方法は燃料を安定して供給しなければなりません。

（Enter）肉の断片、オリンピックのために用意した大量のお弁当オリンピックのために用意した大量のお弁当　　※言わなくても言ってもどっちでもいい。
）などなど日本にはたくさんの食料廃棄物、つまり有機物がゴミとして存在しています。
それをバイオマス発電の燃料として使えば、普通にゴミ焼却施設で燃やしてco2を出すよりはマシなのではないのでしょうか。
この方法を使うことで、食品ロス問題とCO2の排出量の減少のどちらも解決できて、winwinなのではないのでしょうか。

日本の家庭からは年間4000万トン、
企業からは3億4千万トンのゴミが出てきます。
その中の有機ごみはおよそ4000万トンだと仮定すると、バイオマス発電で年間、216億kwh、つまり日本の年間そう発電量の約3~4%発電できます。

次に水素発電です。
水素発電は名前の通り、水素を燃料として利用し、化学反応を通じて電気を生み出す発電方法です。
水素と酸素を化学反応させ、電気を直接生成する燃料電池方式、水素を燃やして、蒸気や高温ガスを活用してタービンを回す水素ガスタービン方式の2種類があります。
水素を燃やす過程では、水だけを排出するのd、CO2フリーのクリーンな発電方法でもあります。
しかし、こちらは課題まみれです。
今の日本の技術だと大規模な水素発電所を建てることはほぼ不可能に近いです。
次に水素発電をするには大量の水素が必要です。
つまり、水素発電には水素輸送網を作り、各地に水素を運ぶ必要があります。
ここで私たちは考えました。
水素を輸入している港があります。
そう、神戸港です。
私はこの神戸港を中心に水素輸送網を引き、水素発電を広げればいいです。
水素発電は一機当たり日本の年間発電量の0.0008%発電できます。
（将来が不安だ）これからどんどん技術も進化するはずなので、大丈夫でしょう。
（全然大丈夫じゃない）
主力として有力なのが原子力発電です。
原子力発電は、核分裂によって熱エネルギーを生み出し、電力を供給する発電方式です。
原子炉内でウランやプルトニウムの原子核が分裂すると、大量の熱が発生します。
この熱を利用して水を沸騰させ、高温の蒸気を作り出します。
その蒸気がタービンを回し、発電機によって電気が生み出されます。
この発電方法は、二酸化炭素（CO₂）の排出量が少なく、長時間安定して大量の電力を供給できることが特徴です。
そのため、電力需要が高い国や地域では重要な電源として利用されています。
しかしながら、放射性廃棄物の処理や、安全性の確保といった課題も抱えています。
特に、過去の原子力事故（チェルノブイリ、福島など）から学び、厳格な安全対策が求められています。
加えて、原子力発電所の建設には莫大な初期投資が必要であり、政策の影響を大きく受けることから、新規の導入や維持には慎重な検討が必要です。
こうした課題を克服するため、技術の進化や安全対策の強化が進められています。
原子力発電一機当たり年間に日本の総発電量の0.8%発電できます。
ちょっと主力としては力不足かもしれませんね。


ですが、次の発電は即戦力の中でもトップクラスの発電量があります。
そう、潮力発電です。
潮力発電は潮の満ち引きや潮流を利用して電力を生み出す発電方法です。
潮の動きは一定なため、安定した発電が可能です。
CO2は皆無で環境負荷がとても低いです。
発電施設は数十年単位で稼働できるため、とても寿命が長いです。
私の班ではこれを瀬戸内海に設置したらいいのではないかと考えました。
ノルウェーのクバルスン海峡を参考に試算した結果、一気当たり年間約5000万kwhも発電が可能です。
めちゃくちゃ簡単にいうと、200台おけば日本の発電量は余裕で超えます。
しかし、設置コストがとても高いです。
小規模で五百〜700万、大規模になると数十億かかります。
定期的なメンテナンスが必要です。

次に紹介するのは、これまた即戦力の代表格。
そう、風力発電です！風力発電はその名の通り、風の力でタービンを回して電力を生み出す方法です。
風さえ吹けば発電できるので、燃料いらず＆CO2ゼロ。
超エコです。
しかも、設置後の運転コストがかなり低く、うまくいけば何十年も稼働可能。
ヨーロッパでは既に主力の再生可能エネルギーになっています。

私たちはこれを北海道の日本海側に設置するのが良いと考えました。
 風が安定して吹く地域だからです。

実際、風速6〜7m/sを超えるエリアでは、1基で年間400万kWh以上発電可能。
簡単に言うと、大型風車を1000基建てれば原発10基分の電力も夢じゃありません！…が、弱点もあります。
 風が弱い日は発電量が落ちるし、騒音問題や景観への影響もあるので、設置場所はよーく考える必要があります。

そして再生可能エネルギーの代表格とも言えるのが、太陽光発電です。
太陽の光をパネルで受け取り、電力へと変換するという、非常にシンプルでありながら高度な仕組みを持つ発電方法です。
二酸化炭素の排出はゼロ、燃料も不要で、運転時に音も出さないため、非常に静かで環境負荷の少ない技術といえます。

日本国内には日照時間が比較的長い地域も多く、住宅の屋根にパネルを設置するだけで発電が可能という手軽さも、大きな利点です。
例えば静岡県のように日照条件が安定している地域では、1枚あたり約1.6㎡のパネルで年間およそ400kWhの電力を生み出すことができるとされています。
これを1戸の住宅に30枚設置した場合、年間で約12,000kWhの発電が可能となり、一般的な家庭であればほぼ自給自足が実現できるレベルです。

もっとも、太陽光発電にも課題は存在します。
天候に左右されやすく、曇天時や降雨時は発電量が減少し、夜間には一切発電できません。
また、近年では使用済みパネルの廃棄・リサイクルに関する問題も注目を集めています。

当初、私たちの班では、バイオマス、原子力、水素を中心とした発電を基盤にし、そこへ水力、風力、太陽光、潮力といった再生可能エネルギーを「即戦力」として組み込むことで、日本の電力需要を十分に賄えると考えていました。

──しかし、現実はそれほど簡単ではありませんでした。

確かに、これらの発電方式を導入すればCO₂排出量の削減にはつながります。
しかしながら、いずれの方法も「十分な電力量を確保するには大規模な導入が必要」「初期導入コストが高い」「設置場所の確保が困難」といった課題に直面しました。

最終的に私たちが導き出した結論は、意外なものでした。

それは── 「現状維持が、最も現実的である」ということです。


……でも、それじゃ「CO₂排出ゼロ」という目標はどこへ行ったんだ？って話になります。
そこで、私たちは発想を逆転しました。
「火力発電をやめる」のではなく、火力発電で出たCO₂を、他で吸収すればいいのでは？現在、CO₂を吸収する装置の開発・研究が世界中で進んでいます。
もしこの装置が、火力発電で排出されるCO₂と同量を吸収できるようになれば、実質的に火力発電のCO₂は“ゼロ”になります。
これは、CO₂を吸収しながら燃やすバイオマス発電と同じ理屈です。
火力発電の効率や排出量を少しずつ改良しつつ、CO₂吸収装置の性能も上げていけば、火力発電をカーボンニュートラル化する道が、現実的になってくるのではないでしょうか。
